# LifeLog 仕様案

## 1. コンセプト

LifeLog は、短いタイトルと短い本文だけで日々の思考や出来事を残していく個人用ライフログサイトです。
本格的なブログやメディアではなく、1 日 1 回程度の独り言を、できるだけ長く、できれば一生運用することを目的にします。

## 2. 目的

- 自分が生きた記録を時系列で残す。
- その時々の考え、感情、気づきを短く保存する。
- 投稿や管理の負担を極力小さくし、長期運用しやすくする。
- GitHub だけで無料運用できる構成を優先する。
- 将来的に記事数が増えても、年別・月別で自然に振り返れるようにする。

## 3. 対象外

以下は初期仕様では扱いません。

- カテゴリ、タグ、シリーズなどの分類機能。
- コメント機能。
- いいね、アクセス解析、SNS 連携などの反応獲得機能。
- 複数ユーザー管理。
- 下書き、予約投稿、管理画面などの CMS 的な機能。
- 画像中心の記事や長文ブログ向けの複雑なレイアウト。

## 4. 基本方針

### 4.1 最小限の投稿単位

1 件の記録は次の情報だけを持ちます。

| 項目 | 必須 | 説明 |
| --- | --- | --- |
| 日付 | 必須 | 記録した日。URL や一覧表示にも使う。 |
| タイトル | 必須 | その日の短い見出し。 |
| 本文 | 必須 | 短い本文。数行から数段落程度を想定する。 |

初期段階ではカテゴリ、タグ、アイキャッチ画像、概要文などは持たせません。

### 4.2 URL 設計

日付をもとに、長期運用しても壊れにくい URL にします。

推奨形式:

```text
/entries/YYYY/MM/DD/
```

例:

```text
/entries/2026/06/09/
```

同じ日に複数投稿したくなった場合は、初期方針としては「その日の本文に追記する」運用にします。
将来的に複数投稿が必要になった場合のみ、`/entries/YYYY/MM/DD-2/` のような拡張を検討します。

### 4.3 一覧ページ

最低限、以下のページを用意します。

| ページ | 役割 |
| --- | --- |
| トップページ | 最新の記録を新しい順に表示する。 |
| 年別ページ | その年の記録を月別に見られるようにする。 |
| 月別ページ | その月の記録を日付順または新しい順に表示する。 |
| 個別ページ | 1 件の記録を表示する。 |

トップページには最新 10〜30 件程度を表示し、過去分は年別・月別リンクから辿れるようにします。

### 4.4 ナビゲーション

サイト全体のナビゲーションは簡素にします。

- トップへ戻るリンク。
- 年別アーカイブへのリンク。
- 個別ページでは前の日記・次の日記へのリンク。
- 月別ページでは前月・次月へのリンク。

検索機能は初期仕様では不要です。記事数が大きく増えた後に必要性を再判断します。

## 5. 投稿ファイルの管理方針

### 5.1 推奨ファイル形式

GitHub 上で直接編集しやすく、静的サイト生成にも向いている Markdown を推奨します。

推奨配置:

```text
entries/YYYY/MM/DD.md
```

例:

```text
entries/2026/06/09.md
```

### 5.2 投稿ファイルの例

```markdown
---
date: 2026-06-09
title: 今日から記録する
---

少しずつ、自分の考えを残していくことにした。
長く続けるために、短くてもいいことにする。
```

Front Matter は `date` と `title` のみに限定します。
本文は Markdown で書けるようにしますが、見出しや装飾を多用しない前提にします。

### 5.3 投稿追加方法

初期運用では、GitHub だけで完結する方法を優先します。

1. GitHub の Web UI で `entries/YYYY/MM/DD.md` を作成する。
2. `date` と `title` を入力する。
3. 本文を書く。
4. Commit する。
5. GitHub Actions が静的サイトを生成し、GitHub Pages に公開する。

ローカル環境や有料サービスがなくても投稿できることを重要視します。

## 6. GitHub Pages 前提の公開方式

このリポジトリでは、外部パッケージに依存しない Node.js 製の静的サイト生成スクリプトで `entries/` から `dist/` を生成します。
GitHub Actions が `dist/` を GitHub Pages に公開します。

### 6.1 ホスティング

初期公開先は GitHub Pages とします。
独自ドメインは任意で、最初は `https://<user>.github.io/<repo>/` 形式でも問題ありません。

### 6.2 ビルド

Markdown の投稿ファイルから静的 HTML を生成し、GitHub Pages に公開します。

初期実装では、依存関係を増やさずに `scripts/build.mjs` で HTML を生成します。
将来的に機能が増えて保守が難しくなった場合のみ、Eleventy、Jekyll、Astro などへの移行を検討します。

## 7. デザイン方針

### 7.1 見た目

- 文字を読むことに集中できる、余白のあるシンプルな画面にする。
- 装飾は最小限にする。
- ダークモード対応は必須ではないが、長期的には検討する。
- スマートフォンで読み書きする前提を強く意識する。

### 7.2 表示情報

個別ページには以下だけを表示します。

- 日付。
- タイトル。
- 本文。
- 前後の記録へのリンク。

著者プロフィール、関連記事、カテゴリ一覧などは表示しません。

## 8. 長期運用のための考慮

### 8.1 データの寿命

投稿の本体は Markdown ファイルとしてリポジトリに保存します。
これにより、将来別の静的サイトジェネレーターへ移行する場合でも、文章データを再利用しやすくします。

### 8.2 URL の寿命

日付ベースの URL は、タイトル変更の影響を受けないため長期運用に向いています。
タイトルを変えても URL は変えない方針にします。

### 8.3 バックアップ

GitHub リポジトリ自体が履歴を持ちますが、長期運用を考えると定期的にローカルや別ストレージへバックアップする運用も検討します。

### 8.4 移行性

特定サービスへの依存を減らすため、投稿データは Markdown と YAML Front Matter のような一般的な形式にします。

## 9. 初期リリースの最小機能

最初の公開版では、以下だけを実装対象とします。

1. Markdown で日記を追加できる。
2. トップページに最新の記録一覧を表示できる。
3. 個別ページで 1 件の記録を読める。
4. 年別・月別アーカイブへ移動できる。
5. GitHub Actions で GitHub Pages に自動公開できる。
6. スマートフォンでも読みやすい最低限のレスポンシブデザインにする。

## 10. 後回しにする機能

以下は必要になったら追加します。

- 全文検索。
- RSS フィード。
- ダークモード。
- 投稿作成を補助する GitHub Actions ワークフロー。
- 月ごとの振り返りページ。
- データのエクスポート機能。
- 独自ドメイン設定。

## 11. 未決定事項

今後決める必要がある項目です。

- 自作の静的サイト生成スクリプトを継続するか、将来どこかで Eleventy、Jekyll、Astro などへ移行するか。
- トップページに表示する件数。
- 年別・月別ページの表示順。
- サイトタイトルとサブタイトル。
- 公開範囲を完全公開にするか、検索避けを意識するか。
- RSS を初期から出すか、後回しにするか。

## 12. 推奨する次のステップ

1. GitHub Pages のリポジトリ設定で Pages の公開元を GitHub Actions にする。
2. サイトタイトルとサブタイトルを自分用に調整する。
3. スマートフォンから GitHub Web UI で投稿を追加する運用を試す。
4. 必要になったら RSS、検索、投稿補助ワークフローなどを追加する。
